# 播放卡顿分析报告

**设备**: 192.168.1.100
**应用**: com.hsvj.engine (MainActivity)
**采集时间**: 2026-06-20 11:25 - 11:27
**视频源**: MIPI 采集 1920x1080 @ 148.5MHz pixel clock
**解码器**: h264_rkmpp (瑞芯微硬件解码)
**视频分辨率**: 2626x2160 (coded 2688x2176)

---

## 一、核心现象

### 1.1 帧率严重低于目标

| 指标 | 目标值 | 实际值 | 差距 |
|------|--------|--------|------|
| 目标帧间隔 | 16ms (60Hz) | - | - |
| 实际帧率 (actualFps) | 60 FPS | **8.8 - 10.9 FPS** | 下降 82%-85% |
| 平均更新耗时 (avgUpdateMs) | <16ms | **90.7 - 113.0ms** | 超标 5.7-7.1 倍 |
| 最大更新耗时 (maxUpdateMs) | <16ms | **91 - 119ms** | 超标 5.7-7.4 倍 |
| 平均总耗时 (avgWallMs) | <16ms | **90.8 - 113.2ms** | 超标 5.7-7.1 倍 |

### 1.2 帧率持续恶化趋势

采样时间序列显示帧率在下降，卡顿逐步加重：

| 时间 | actualFps | avgUpdateMs | maxUpdateMs |
|------|-----------|-------------|-------------|
| 10:56:32 | 10.9 | 90.7 | 91 |
| 10:56:34 | 10.9 | 90.7 | 91 |
| 10:56:36 | 10.9 | 90.8 | 94 |
| 11:26:05 | 8.9 | 111.6 | 114 |
| 11:26:21 | 8.8 | 112.7 | 117 |
| 11:26:23 | 8.8 | 113.0 | 119 |
| 11:26:25 | 8.9 | 111.9 | 115 |
| 11:26:29 | 8.8 | 112.2 | 119 |
| 11:26:31 | 8.8 | 112.8 | 118 |

**结论**: 30 分钟内帧率从 10.9 下降到 8.8 FPS，avgUpdateMs 从 90ms 上升到 113ms，存在性能持续退化（可能是内存泄漏或资源未释放）。

---

## 二、根因分析

### 2.1 根因一：音频错误日志刷屏（最严重）

**现象**:
- 错误日志 `AudioHardwareTiny: getInputRouteFromDevice:device:80000020` 以约 **100次/秒** 的频率刷屏
- 在 10000 条 logcat 采样中，该错误占 **4941 条（49.4%）**
- 错误来源：audioserver (PID 219)

**影响**:
- audioserver 进程占用 **23.3% CPU**（正常应 <5%）
- logd 进程占用 **13.3% CPU** 处理日志写入
- 大量 IO 和 CPU 资源被日志系统占用，挤压视频渲染线程
- 设备 80000020 对应 `AUDIO_DEVICE_IN_HDMI`，表示 HDMI 音频输入路由配置失败

**根因**: HDMI 音频输入设备路由异常，音频 HAL 层在循环尝试获取输入路由但持续失败，未做错误抑制（rate limit）。

### 2.2 根因二：MIPI 采集层丢帧

**现象**:
- 内核日志 `rkcif_mipi_lvds: not active buffer, skip current frame, mipi/lvds stream[0]` 持续出现
- 采集驱动频繁因"无可用缓冲区"跳过当前帧

**影响**:
- 采集端帧丢失，视频源数据不完整
- 解码器接收到的数据流可能存在时间戳跳跃，导致渲染端等待或丢帧

**根因**: V4L2 采集缓冲区队列不足，或用户空间消费缓冲区速度跟不上采集速度，导致驱动层无活跃缓冲区可用。

### 2.3 根因三：解码与渲染速率不匹配

**现象**:
- 解码器诊断 `[RkmppPacketDiag]` 显示：
  - 11:26:40 count=27300
  - 11:26:50 count=27600
  - **10 秒解码 300 帧 = 30 FPS 解码**
- 但实际渲染帧率仅 **8.8 FPS**
- 解码出的帧约 **70% 被丢弃**（30 解码 - 8.8 渲染 = 21.2 帧丢弃/秒）

**影响**:
- 解码器满负荷工作但产出被浪费
- 解码线程占用 CPU/GPU 资源，进一步挤压渲染线程

**根因**: 渲染循环每帧耗时 113ms（远超 33ms 的 30FPS 帧间隔），无法消费解码器产出的帧，导致帧丢弃。

### 2.4 根因四：CPU 资源紧张

**系统 CPU 负载**:
```
400%cpu  53%user  7%nice  93%sys  227%idle  10%irq  10%sirq
```
- 总使用率 **173%**（4核中约 1.7 核忙碌）
- **sys 占用 93%**（内核态高，异常）
- irq + sirq 占 20%（中断处理开销大）

**进程 CPU 占用**:
| 进程 | CPU% | 说明 |
|------|------|------|
| com.hsvj.engine | 63.3% | 主应用，单核已饱和 |
| audioserver | 23.3% | 异常，音频错误刷屏导致 |
| top | 20.0% | 采样工具 |
| adbd | 20.0% | ADB 调试 |
| logd | 13.3% | 日志系统，被音频错误拖累 |
| surfaceflinger | 10.0% | 合成器 |

**系统负载**: loadavg 2.74 2.38 1.89（4核系统，负载偏高）

**根因**: com.hsvj.engine 单进程占用 63% CPU 已接近单核上限，加上 audioserver 异常占用，系统 CPU 资源紧张，渲染线程无法获得足够时间片。

### 2.5 根因五：视频分辨率与码率偏高

**视频参数**:
- 分辨率：2626x2160（约 568 万像素，接近 4K）
- 单帧码率：120-153 KB/帧
- 估算总码率：约 30-45 Mbps

**影响**:
- 高分辨率导致解码、渲染、合成各环节耗时增加
- 在 4 核 RK 平台上处理 4K 级别视频压力较大

## 三、卡顿链路图

```
[MIPI 采集 1920x1080]
        │
        ▼ (缓冲区不足，丢帧)
[V4L2 驱动] ──> "not active buffer" 丢帧
        │
        ▼
[h264_rkmpp 解码] ──> 30 FPS 解码（满负荷）
        │
        ▼ (渲染消费慢，70% 帧丢弃)
[渲染循环] ──> 8.8 FPS 渲染（目标 60 FPS）
        │     │
        │     └──> avgUpdateMs=113ms（应 <16ms）
        ▼
[SurfaceFlinger 合成] ──> 60Hz 刷新（帧重复显示）

并行问题：
[audioserver] ──> 音频路由错误 100次/秒刷屏 ──> 占用 23% CPU + logd 13% CPU
```

## 四、优化建议

### 4.1 优先级 P0：修复音频错误刷屏

**问题**: `AudioHardwareTiny: getInputRouteFromDevice:device:80000020` 错误刷屏

**建议**:
1. 检查 HDMI 音频输入设备配置，设备号 80000020 = `AUDIO_DEVICE_IN_HDMI`
2. 确认 HDMI 是否实际连接并支持音频输入
3. 若不需要 HDMI 音频输入，在音频策略配置中移除该设备路由
4. 在 HAL 层增加错误抑制（rate limit），避免相同错误每秒刷 100 次
5. 预期收益：释放 audioserver 23% CPU + logd 13% CPU = **36% CPU 资源**

### 4.2 优先级 P0：增加 MIPI 采集缓冲区

**问题**: `not active buffer, skip current frame`

**建议**:
1. 增加 V4L2 采集缓冲区数量（建议从默认 4 增加到 8-12）
2. 检查用户空间缓冲区消费线程是否存在阻塞
3. 确认采集线程优先级是否足够（建议 SCHED_FIFO 实时调度）
4. 预期收益：减少采集端丢帧，保证视频源数据完整

### 4.3 优先级 P1：优化渲染循环

**问题**: avgUpdateMs=113ms，远超 16ms 目标

**建议**:
1. 分析 `avgUpdateMs` 的构成：是解码等待、格式转换、还是绘制本身？
2. 若是解码等待，考虑异步解码 + 帧队列解耦
3. 若是格式转换（YUV→RGB），使用 GPU 着色器替代 CPU 转换
4. 若是绘制耗时，检查是否有不必要的全屏重绘
5. 降低渲染分辨率：2626x2160 → 1920x1080，减少 65% 像素处理量

### 4.4 优先级 P1：解码渲染同步策略

**问题**: 解码 30 FPS 但渲染 8.8 FPS，70% 帧浪费

**建议**:
1. 实现帧丢弃策略：当渲染落后时，解码器只保留最新帧 + 关键帧
2. 避免 B 帧重排序开销：渲染慢时跳过 B 帧解码
3. 考虑使用 Surface 直接渲染（SurfaceView + MediaCodec），跳过 CPU 拷贝

### 4.5 优先级 P2：降低视频源负载

**问题**: 2626x2160 分辨率偏高

**建议**:
1. 若采集源支持，降低输出分辨率到 1920x1080
2. 若必须高分辨率采集，在送解码前先硬件缩放
3. 降低码率：当前 30-45 Mbps 偏高，建议 15-20 Mbps

### 4.6 优先级 P2：性能持续退化排查

**问题**: 30 分钟内帧率从 10.9 降到 8.8 FPS

**建议**:
1. 监控 Native Heap 增长（当前 33MB，观察是否持续增长）
2. 检查是否有缓冲区未释放（V4L2 buffer leak）
3. 检查 GPU 内存（当前 4.05MB，观察是否增长）
4. 长时间运行测试（2 小时+）观察内存趋势

---

## 五、预期优化效果

| 优化项 | 预期 CPU 释放 | 预期帧率提升 |
|--------|---------------|--------------|
| 修复音频刷屏 | 36% | +5-8 FPS |
| 增加采集缓冲区 | - | +2-3 FPS（减少丢帧） |
| 优化渲染循环 | - | +10-15 FPS |
| 解码渲染同步 | 10-15% | +3-5 FPS |
| 降低分辨率 | 20-30% | +8-12 FPS |

**综合预期**: 优化后帧率可达 **30-45 FPS**，基本满足流畅播放需求。
