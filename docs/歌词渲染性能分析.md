# 歌词视频播放卡顿 - 性能分析报告

## 一、调用链概览

```
Engine::renderFrame()
  ├─ beginFrame()
  ├─ for (可见图层) updateTexture() / updateVideoTexture()
  │     └─ Layer 21: updateLayer21Texture()
  │           └─ lyricRenderer_->prepareFrame()  ← 每 2 帧调用一次
  │                 ├─ ass_render_frame()        ← 【瓶颈 1】CPU 重
  │                 └─ buildGpuMaskFrame()
  │                       └─ endMaskTextureWrite() → vkQueueWaitIdle()  ← 【瓶颈 2】GPU 同步
  ├─ beginCanvasRenderPass()
  └─ renderLayersToCanvas()
        └─ for (renderQueue) item.layer->render() / renderSliceItem()
              └─ Layer 21: lyricRenderer_->renderFrame()  ← 每 LAYER + 每 SLICE 各调用一次
                    └─ renderAssMaskBatch()     ← 【瓶颈 3】重复绘制
```

---

## 二、已确认的性能瓶颈

### 瓶颈 1：ass_render_frame 主线程同步（CPU 重）

**位置**：`ASSRenderer::renderFrame()` → `ass_render_frame()`

**现象**：
- libass 在主线程同步执行字体光栅化、布局、字形合成
- 注释已注明：`libass 单次 60ms+ 时需降低`（LyricRenderer.cpp:244）
- 当前已通过 `s_lyricFramePhase` 将 prepareFrame 降为约 30fps，但每帧仍可能 10–30ms+

**影响**：主线程阻塞，直接导致掉帧、卡顿。

---

### 瓶颈 2：mask 纹理上传使用 vkQueueWaitIdle（GPU 同步）

**位置**：`VulkanRenderer_AssMask.cpp` → `endMaskTextureWrite()` / `endMaskTextureWriteRows()`

```cpp
vkQueueSubmit(graphicsQueue_, 1, &submitInfo, VK_NULL_HANDLE);
vkQueueWaitIdle(graphicsQueue_);  // 主线程等待 GPU 完成
```

**现象**：
- 每次 mask atlas 上传（全量或脏行）都通过 `beginSingleTimeCommands` 提交
- `endSingleTimeCommands` 内部调用 `vkQueueWaitIdle`，主线程阻塞直到 GPU 完成
- 在 prepareFrame 阶段执行，阻塞整帧渲染

**影响**：GPU 与 CPU 强同步，增加帧延迟，易造成卡顿。

---

### 瓶颈 3：Layer 21 主图 + 切片重复绘制

**位置**：`Engine_Render.cpp` → `renderLayersToCanvas()`

**现象**：
- Layer 71（二维码）有特殊逻辑：仅渲染切片，不渲染主图
- Layer 21 无此逻辑：主图 + 所有切片都会加入 renderQueue
- 若 Layer 21 配置了多个图层切片（切片是图层的子配置，用 coordinate/range 在幕布上定义该图层的显示坐标，与输出矩阵无关），则同一歌词内容会被绘制 1 次主图 + N 次切片
- 每次 `renderAssMaskBatch` 都会：`vkMapMemory` → 填充 instance → `vkUnmapMemory` → `vkCmdDraw`

**影响**：同一内容多次绘制，增加 CPU 与 GPU 负担。

---

### 瓶颈 4：buildGpuMaskFrame 的 CPU 开销

**位置**：`LyricRenderer_Gpu.cpp` → `buildGpuMaskFrame()`

**现象**：
- `packMaskAtlas`：bin packing 算法，O(n) 遍历
- `blitMaskWithBorder`：每个字形一次 memcpy + 边框扩展
- 脏行检测：整图 memcmp
- 字形数量多时（如复杂 K 歌效果），CPU 时间可观

**影响**：在 prepareFrame 阶段占用主线程，叠加 ass_render_frame 后更易超时。

---

### ~~瓶颈 5：锁竞争~~（已排除）

**说明**：Layer 21 独享一套 libass 上下文，所有 lyric 操作（prepareFrame、renderFrame、load、setASSStyle）均通过同一实例。若这些调用都在**主线程**顺序执行（典型渲染循环），则不存在多线程同时抢锁，**无锁竞争**。锁仅用于防御性线程安全，开销可忽略。

## 三、优化建议（按优先级）

### 高优先级

| 优化项 | 说明 | 预期收益 |
|--------|------|----------|
| **1. Layer 21 有切片时跳过主图** | 仿照 Layer 71，当 Layer 21 存在切片时只渲染切片，不渲染主图 | 减少 1 次重复 draw，降低 CPU/GPU 负载 ✅ 已实施 |
| **2. 异步 mask 纹理上传** | 将 `endMaskTextureWrite` 中的 `vkQueueWaitIdle` 改为使用当前帧 command buffer 记录 | 消除 prepareFrame 中的 GPU 同步阻塞 ✅ 已实施 |
| **3. 降低 libass 调用频率** | 将 `s_lyricFramePhase` 从 2 帧改为 3 帧（约 20fps） | 进一步降低 ass_render_frame 调用次数 ✅ 已实施 |

### 中优先级

| 优化项 | 说明 | 预期收益 |
|--------|------|----------|
| **4. 降低歌词渲染分辨率** | 将 `kLyricMaxWidth/Height` 从 960×540 降到 800×450 | 减少 libass 与 atlas 计算量 ✅ 已实施 |
| **5. 合并多切片为单次 draw** | 当 Layer 21 有多个幕布坐标切片时，在单次 `renderAssMaskBatch` 中按各切片区域生成 instance，减少 draw 次数 | 减少 vkMap/vkUnmap 和 draw call ✅ 已实施 |

### 低优先级

| 优化项 | 说明 | 预期收益 |
|--------|------|----------|
| **6. libass 后台线程** | 将 ass_render_frame 移到后台线程（需解决 ASS_Image 所有权与 ass_frame_unref 的线程安全） | 减轻主线程 CPU 负载，实现难度高 |

---

## 四、建议实施顺序

1. **立即实施**：优化 1（Layer 21 有切片时跳过主图）——改动小、风险低、收益明确。
2. **短期**：优化 2（异步 mask 上传）——需调整 Vulkan 上传流程，收益大。
3. **短期**：优化 3（降低 libass 频率）——简单参数调整。
4. **中期**：优化 4、5——视实际 profiling 结果再决定投入。

## 五、概念澄清：切片 vs 输出区域

| 概念 | 职责 | 关系 |
|------|------|------|
| **切片 (slice)** | 图层子配置，用 coordinate/range 在**幕布**上定义该图层的显示坐标 | 决定图层在画布哪块绘制 |
| **输出区域 (output region)** | 由 output_layout 矩阵管理，**所有图层合成后**的画布统一经矩阵分割、旋转、输出 | 与切片无关，是渲染末端的显示管理 |

流程：各图层（含切片）→ 合成到幕布 → 矩阵输出。

## 七、内存优化（2025-03 补充）

**现象**：开歌词 483MB vs 不开 130MB，歌词相关约 353MB；此前约 194MB。

**已实施**：
- libass 默认值（ass_render.h）：字形 2000、位图 8MB（原 10000/128MB）
- mask atlas 最大 2048×2048（原 4096×4096，单 atlas 16MB）
- staging buffer 可 shrink：当 atlas 缩小至不足 1/4 时回收，避免只增不减

**排查建议**：若仍偏高，检查生产 ASS 是否含 `[Fonts]` 内嵌字体（`ass_set_extract_fonts(0)` 已禁用提取）。
