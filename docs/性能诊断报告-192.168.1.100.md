# 性能诊断报告 - 192.168.1.100

## 设备信息
- **设备型号**: rk3566_r
- **平台**: rk356x (RK3566)
- **内存**: 2GB (可用仅180MB，压力大)
- **应用进程**: com.hsvj.engine (PID: 4453)

## 核心问题诊断

### 🔴 问题1：采集图层DMA导入严重阻塞（最严重）

**现象**：
```
renderFrame_total cost=46.93ms (目标<16ms才能60fps)
  - capture更新: 38.09ms  ← 占用82%时间
  - beginFrame: 3.41ms
  - present: 3.78ms
```

**详细瓶颈**：
```
dma_buf_rebind_waitIdle: 30ms  ← GPU等待空闲
dma_buf_import: 32ms           ← DMA buffer导入GPU
```

**原因分析**：
1. MIPI采集使用DMA buffer直通（zero-copy），但每帧都需要：
   - 等待GPU完成前一帧的渲染（waitIdle）
   - 将新的DMA buffer导入到Vulkan纹理（vkBindImageMemory）
   
2. RK3566的GPU（Mali-G52）处理1920x1080 NV12格式导入较慢

3. **内存压力大**（可用仅180MB），导致DMA分配/绑定变慢

**影响**：
- 理论帧率：1000ms / 46.93ms ≈ **21fps**
- 实际会更低，因为还有其他开销

---

### 🟡 问题2：media.swcodec进程占用91% CPU

**现象**：
```
91% 9535/media.swcodec: 89% user + 2.1% kernel
29% 4453/com.hsvj.engine: 18% user + 11% kernel
```

**原因**：
- VP8编码器在工作（不是解码器！）
- 从崩溃堆栈看：`vp8e_encode` → 这是**编码**过程
- 可能是某个功能在录制或推流

**诊断**：不是播放卡顿的主要原因，但会加重CPU负担

---

### 🟢 问题3：视频分辨率过高

**发现**：
```
Adjusted frame pool size: video=2626x2160 heavy=1
```

**分析**：
- 视频分辨率：2626×2160（超过4K）
- 在2GB内存的RK3566上处理4K视频压力极大
- 与采集的1920×1080叠加渲染，GPU负担翻倍

---

### 🟡 问题4：内存压力

```
MemTotal: 2003164 kB (1.9GB)
MemAvailable: 179804 kB (175MB)  ← 非常紧张
Native Heap: 38MB
```

**影响**：
- DMA buffer分配变慢
- 频繁的内存回收
- GPU内存不足导致频繁的纹理重建

---

## 解决方案

### 🚨 立即实施（效果最明显）

#### 方案A：优化采集图层的DMA导入（代码级）

**当前问题**：每帧都同步等待GPU空闲再导入DMA buffer

**优化代码位置**：`src/renderer/VulkanRenderer_Texture.cpp`

```cpp
// 当前实现（推测）
void VulkanRenderer::importDmaBuffer(...) {
    vkQueueWaitIdle(graphicsQueue_);  // ← 30ms阻塞！
    vkBindImageMemory2(...);           // ← 32ms阻塞！
}

// 优化方案1：使用Fence异步等待
void VulkanRenderer::importDmaBufferAsync(...) {
    // 不要waitIdle，使用fence
    VkFence fence = getOrCreateFence();
    vkQueueSubmit(..., fence);  // 提交前一帧
    
    // 在需要时才等待（而不是立即）
    // 让CPU继续准备其他内容
}

// 优化方案2：双缓冲DMA texture
class CaptureTexturePool {
    VkImage textures[3];  // 3个纹理轮换
    int currentIndex = 0;
    
    VkImage getNextTexture() {
        // 轮换使用，避免等待前一帧GPU完成
        currentIndex = (currentIndex + 1) % 3;
        return textures[currentIndex];
    }
};
```

**预期收益**：采集时间从38ms → **5-10ms**，帧率提升到50-60fps

### 📋 中期优化（需要代码修改）

#### 方案D：异步采集纹理更新

修改 `LayerVideo::updateCaptureTexture()` 实现：

```cpp
// 文件：src/layer/LayerVideo_Capture.cpp

void LayerVideo::updateCaptureTexture() {
    // 当前：同步等待新帧 → 同步导入GPU
    
    // 优化：后台线程准备，主线程只取结果
    if (!captureTextureFuture_.valid()) {
        // 异步开始准备下一帧
        captureTextureFuture_ = std::async(std::launch::async, [this]() {
            return captureDmaBufferAsync();
        });
    }
    
    if (captureTextureFuture_.wait_for(0) == std::future_status::ready) {
        // 新帧准备好了，快速切换
        currentCaptureTexture_ = captureTextureFuture_.get();
    }
    // 否则继续使用上一帧（允许偶发跳帧，但保证流畅）
}
```

**预期收益**：采集不阻塞主渲染循环，稳定60fps

## 性能监控脚本

保存为 `monitor_performance.sh`：

```bash
#!/bin/bash

echo "开始监控性能..."
adb logcat -c

while true; do
    echo "=== $(date) ==="
    
    # CPU使用率
    echo "CPU使用率："
    adb shell "top -n 1 -m 5 | grep -E 'media.swcodec|hsvj.engine'"
    
    # 内存
    echo "可用内存："
    adb shell "cat /proc/meminfo | grep MemAvailable"
    
    # 最近的渲染性能
    echo "渲染性能："
    adb logcat -d -t 20 | grep "SwitchStall\|renderFrame_total" | tail -3
    
    echo ""
    sleep 5
done
```

运行：
```bash
bash monitor_performance.sh
```

## 总结

**根本原因**：
1. **采集图层DMA导入同步阻塞**（38ms）← 主要瓶颈
2. 4K视频与1080p采集同时渲染，GPU和内存压力过大
3. 系统可用内存仅175MB，严重不足

1. ✅ 关闭采集图层测试（立即验证问题）

3. ✅ 清理内存并重启应用
4. 🔧 实施DMA异步导入优化（代码修改，根本解决）
