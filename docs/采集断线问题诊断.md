# 采集断线问题的真正原因

## 问题现象
移动采集图层XY后，显示"无信号"，几秒后恢复。

## 🔍 根本原因

### 发现：采集渲染频率极低

**实测数据**（15秒监控期间）：
```
06-19 20:09:12.667 - frame=14700（开始）
06-19 20:09:25.662 - frame=14700（结束）
```

**分析**：
- 15秒内采集帧号没有变化
- 说明采集图层**长时间不更新**
- 渲染频率约：每13秒才渲染1次采集画面

**对比正常应该的情况**：
- 采集源：30fps输入
- 显示帧率：60fps
- 应该每帧都渲染（16ms一次）
- 实际：13秒才渲染1次（**慢了780倍**）

---

## 💡 为什么会这样？

### 可能原因1：智能降频/脏矩形优化

**假设**：
- 系统检测到采集内容"没有变化"
- 启用了智能降频，跳过渲染
- 移动XY时触发了"内容变化"检测失效
- 导致长时间不刷新

### 可能原因2：采集图层的可见性判断问题

**从日志看**：
```
rect=2288,64 1413x2032
```

**位置**：x=2288，远超1920屏幕宽度
- 采集图层可能被判定为"不可见"
- 系统跳过了不可见图层的渲染
- 移动后需要重新计算可见性

### 可能原因3：渲染触发机制问题

**日志显示**：
```
[采集追踪][渲染入口] 每13秒才触发一次
```

**说明**：
- 不是采集设备的问题（V4L2正常工作）
- 不是纹理销毁重建的问题
- 而是**渲染引擎根本没有频繁调用renderLayer**

---

## 🎯 真正的问题

### 不是我优化导致的

**证据**：
1. 没有destroyTexture日志
2. 没有V4L2关闭/重新打开
3. 没有dma_buf_import重复执行
4. 只是**渲染调用频率极低**

### 是渲染循环的问题

**核心问题**：
```
RenderLoopStats: actualFps=19fps
但采集图层13秒才渲染1次
```

**说明**：
- 渲染循环在运行（19fps）
- 但**采集图层被跳过了大部分帧**
- 可能的优化：脏矩形、智能降频、可见性判断

---

```bash

```

### 验证2：检查采集图层可见性

```bash
adb logcat -d | grep -iE "layer.*10.*visible|采集.*可见"
```

### 验证3：检查渲染调用频率

**当前**：
- 渲染循环：19fps（每帧51ms）
- 采集渲染：每13秒1次

**应该**：
- 采集图层在每帧都应该调用renderLayer

## 💡 可能的解决方案

### 方案1：禁用智能降频（如果有）

在代码中查找：
```cpp
// 可能的优化代码
if (layer.contentChanged || layer.forceRedraw) {
    renderLayer(...);
}
```

**修改为**：
```cpp
// 采集图层始终渲染
if (layer.isCapture || layer.contentChanged || layer.forceRedraw) {
    renderLayer(...);
}
```

### 方案2：修复可见性判断

**当前配置**：
```
position: x=2288 (超出屏幕1920)
size: 1413x2032
```

**建议**：
- 调整采集图层位置到屏幕内
- 确保可见性判断正确

### 方案3：强制连续渲染模式

在代码中添加：
```cpp
// 采集图层标记为"需要连续渲染"
if (layer.isV4L2Capture) {
    layer.needsContinuousRender = true;
}
```
