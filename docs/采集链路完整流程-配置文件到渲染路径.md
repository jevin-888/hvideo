## 🔍 三种采集类型的完整链路对比

| 阶段 | HDMI | MIPI | USB |
|------|------|------|-----|
| **配置文件** | `captureType: "HDMI"` | `captureType: "MIPI"` | `captureType: "USB"` |
| **启动函数** | `startHdmiCapture()` | `startMipiCapture()` | `startUsbCapture()` |
| **设备检测** | `findHDMIRXDevice()` | `findMIPIDevice(index)` | `findUSBCameraDevice()` |
| **驱动匹配** | `driver: "rk_hdmirx"` | `name: "rkcif_mipi*"` | `driver: "uvcvideo"` |
| **采集类** | `V4L2Capture` | `V4L2Capture` | `UsbCapture` |
| **像素格式** | 自动检测 (NV12/NV16/NV24/YUYV) | 自动检测 (NV12/NV16/NV24/YUYV) | 固定 MJPEG |
| **MIPI链路** | ❌ | ✅ `setupMIPIPipeline()` | ❌ |
| **缓冲区** | DMA-BUF | DMA-BUF | MMAP |
| **渲染类** | `CaptureRenderer` | `CaptureRenderer` | `UsbCaptureRenderer` |
| **解码器** | 无 (YCbCr直接采样) | 无 (YCbCr直接采样) | RKMPP MJPEG 硬解码 |
| **渲染管线** | v4l2Pipeline (YCbCr) | v4l2Pipeline (YCbCr) | drmPrimePipeline |
| **色彩空间** | BT.601 或 BT.709 (自动) | BT.601 或 BT.709 (自动) | N/A (解码为RGB) |
| **缓冲数** | 4 | 6 (优化减少丢帧) | 8 |
| **Audio DSP** | ✅ (HDMI音频直通) | ✅ (MIPI音频) | ❌ (USB Audio由AudioFlinger接管) |

## 🎯 关键设计亮点

### 1. **三条独立通路，互不干扰**
- HDMI/MIPI 和 USB 完全分离，不再共享同一个 `startCapture` 大函数
- 各自有独立的启动函数、采集类、渲染类
- 代码更清晰，维护更容易

### 2. **智能格式检测 (HDMI/MIPI)**
- `pixelFormat: 0` 表示自动检测
- 枚举驱动支持的格式，选择最佳匹配
- 支持 NV12 (4:2:0), NV16 (4:2:2), NV24 (4:4:4), YUYV, UYVY, BGR3
- 保持原始格式一路 DMA-BUF 传递到 Vulkan

### 3. **零拷贝优先 (HDMI/MIPI)**
- 使用 V4L2 multi-plane + DMA-BUF 导出
- 直接传递给 Vulkan 导入 YCbCr 纹理
- 使用 `v4l2Pipeline` 和 YCbCr 采样器
- 整个链路没有 CPU 拷贝

### 4. **RKMPP 硬解码 (USB)**
- USB 摄像头输出 MJPEG (USB带宽限制)
- 使用 RK3568 的 MPP 硬解码 MJPEG
- 解码后得到 DMA-BUF，然后给 Vulkan
- 解码和渲染在不同线程，异步处理
- 三段独立线程：采集 → 解码 → 渲染

### 5. **Buffer 所有权管理 (CaptureRenderer)**
- `markBufferOwned(bufferIndex)` 标记采集缓冲已被渲染器持有
- `safeReleaseFrame(bufferIndex)` 安全释放缓冲回驱动
- `deferredReleaseQueue` 延迟释放队列，避免 GPU 还在使用时归还
- 彻底解决 EINVAL 风暴问题

### 6. **延迟启动采集机制**
- Engine 初始化 Step 4.8：根据 config 延迟启动采集（延迟 3000ms）
- 仅对图层 10/11，且 `captureEnabled=true` 时启动
- 延迟启动避免与其他初始化步骤冲突
- 调用 `checkAndAutoCapture()` 自动选择采集类型并启动

### 7. **UsbCaptureRenderer 三线程架构**
```
采集线程 → pendingMjpeg_ → 解码线程 → readyFrame_ → 渲染线程
         (copy)             (decode)            (dma import)
```
- 单槽队列，永远只持有最新一帧
- 解码和渲染异步，降低时延
- MppFrame 生命周期与纹理 slot 绑定

## 🎯 关键数据结构

### V4L2CaptureConfig
```cpp
struct V4L2CaptureConfig {
  std::string devicePath;      // "/dev/video0"
  int width;                   // 1920
  int height;                  // 1080
  int bufferCount;             // 4 (HDMI) / 6 (MIPI)
  uint32_t pixelFormat;        // 0 = auto
  bool setupMipiPipeline;      // false (HDMI) / true (MIPI)
};
```

### UsbCaptureConfig
```cpp
struct UsbCaptureConfig {
  std::string devicePath;      // "/dev/video20"
  int width;                   // 1920
  int height;                  // 1080
  int bufferCount;             // 8
};
```

### CaptureFrame
```cpp
struct CaptureFrame {
  void* data;                  // mmap 地址 (仅用于非DMA-BUF)
  int dmaBufFd;               // DMA-BUF fd (HDMI/MIPI)
  int width;                  // 真实宽度
  int height;                 // 真实高度
  int stride;                 // 真实 stride
  uint32_t format;           // NV12/NV16/NV24/YUYV/UYVY/BGR3
  int bufferIndex;            // 缓冲索引
  uint64_t sequence;         // 帧序号
};
```

### UsbCaptureFrame
```cpp
struct UsbCaptureFrame {
  void* data;                  // mmap MJPEG 数据
  int size;                   // MJPEG 大小
  int width;                  // 宽
  int height;                 // 高
  int bufferIndex;            // 缓冲索引
  uint64_t sequence;         // 帧序号
};
```
