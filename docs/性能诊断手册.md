# 性能诊断手册 - 针对不流畅问题

## 问题描述
- **配置**：1个视频图层 + 1个采集图层 + 1个提示消息图层
- **现象**：播放不流畅
- **设备**：192.168.1.100

## 二、常见性能问题排查

### 问题A：CPU过载

**症状**：CPU使用率 > 80%

**可能原因**：
1. ✓ 视频软解（未使用硬件加速）
2. ✓ libass歌词渲染（即使Layer 21不可见）
3. ✓ 音频处理占用过高
4. ✓ 后台任务（云同步、素材扫描）

**诊断命令**：
```bash
# 查看CPU各核心使用率
adb shell top -m 10 -s cpu

# 查看解码器类型
adb logcat | grep -E "使用.*解码器|decoder.*type|RKMPP|software"

# 查看是否有歌词在后台渲染
adb logcat | grep -E "Layer.*21|lyric|libass|prepareFrame"
```

**解决方案**：
```bash
# 检查硬解是否启用
curl http://192.168.1.100/api/system/config | grep -i "hwdec\|hardware"

# 如果是MPEG2/MPEG-PS，确认是否强制软解
# 修改配置（如果支持）
```

### 问题B：GPU瓶颈

**症状**：渲染时间 > 16ms（无法达到60fps）

**诊断**：
```bash
# 查看渲染时间统计
adb logcat | grep -E "renderFrame|帧时间|frame.*time"

# 查看GPU错误
adb logcat | grep -E "VkResult|vkQueue|Vulkan.*error"

# 查看是否频繁重建资源
adb logcat | grep -E "createTexture|allocate|rebuild"
```

**可能原因**：
1. 分辨率过高（4K输出但硬件不支持）
2. 纹理频繁创建/销毁
3. 过多的draw call（切片过多）
4. GPU内存不足

### 问题C：内存不足

**症状**：系统变慢、偶发崩溃

**诊断**：
```bash
# 查看内存使用
adb shell dumpsys meminfo <进程名>

# 查看OOM日志
adb logcat | grep -E "OutOfMemory|OOM|bad_alloc"

# 查看GPU内存
adb logcat | grep -E "VkDeviceMemory|vkAllocate.*failed"
```

**可能原因**：
1. 视频缓冲过多
2. 纹理未及时释放
3. 歌词渲染内存泄漏（已知问题，见性能分析文档）

### 问题D：采集信号不稳定

**症状**：采集图层卡顿、有绿屏

**诊断**：
```bash
# 查看采集状态
adb logcat | grep -E "UsbCapture|negotiated.*fps|采集.*帧率"

# 查看是否频繁重连
adb logcat | grep -E "disconnected|reconnect|restart.*capture"

# 查看信号丢失
adb logcat | grep -E "no.*signal|信号丢失|capture.*timeout"
```

**可能原因**：
1. USB带宽不足（多个USB设备）
2. 采集卡不稳定
3. 信号源分辨率/帧率不匹配
4. USB 2.0限制（MJPEG 1080p60需要USB 3.0）

### 问题E：视频解码跟不上

**症状**：视频播放卡顿、掉帧

**诊断**：
```bash
# 查看解码统计
adb logcat | grep -E "decoder.*fps|解码.*帧率|丢帧|drop.*frame"

# 查看是否频繁seek
adb logcat | grep -E "seek|跳转|drift"

# 查看音视频同步
adb logcat | grep -E "audio.*sync|pts.*diff|drift"
```

**可能原因**：
1. 视频码率过高（>20Mbps在某些设备上）
2. 编码格式不支持硬解（VP9、AV1）
3. 硬件解码器失败回退软解
4. 音频同步导致丢帧

## 三、性能分析工具

### 工具1：内置性能日志

在日志中搜索这些关键指标：

```bash
adb logcat | grep -A 5 "性能统计\|Performance\|渲染统计"
```

查找类似这样的输出：
```
[RenderLoop] 性能统计: 
  beginFrame: 1.2ms
  纹理更新: 3.5ms
  Canvas渲染: 8.3ms
  合成输出: 2.1ms
  总计: 15.1ms (66fps理论)
```

### 工具2：Systrace抓取（如果支持）

```bash
# 抓取10秒的系统trace

sleep 10
adb shell atrace --async_stop > trace.txt

# 在Chrome中打开 chrome://tracing 加载trace.txt
```

## 四、临时应急方案

### 方案1：降低分辨率

```http
POST http://192.168.1.100/api/system/resolution
{
  "width": 1280,
  "height": 720
}
```

### 方案2：降低视频图层输出分辨率

通过Web界面：
1. 进入图层管理
2. 选择Layer 1
3. 调整尺寸为 1280x720 或更小

### 方案3：禁用不必要的特效

检查是否启用了：
- 高斯模糊（gaussianBlur > 0）
- 形状遮罩（shapeType != 0）
- 图像反转（invert）
- 裁剪（cropEnabled）

这些都会增加GPU负担。

### 方案4：减少采集帧率

如果采集是60fps，尝试降到30fps：
- 检查采集配置中的captureFps设置
- 或在采集源端限制输出帧率

### 方案5：关闭Layer 41（临时测试）

```http
POST http://192.168.1.100/api/layer/41/visible
{
  "visible": false
}
```

如果关闭后流畅，说明Layer 41的渲染有问题。

## 七、已知性能陷阱

基于代码分析，这些配置会严重影响性能：

### 陷阱1：隐藏的歌词图层仍在渲染
即使Layer 21不可见，如果配置了bindLayerId，可能仍在后台渲染歌词。

**检查**：
```bash
curl http://192.168.1.100/api/layers | jq '.[] | select(.id==21)'
```

**解决**：确保Layer 21的visible为false且没有绑定视频。

### 陷阱2：切片漫游动画
任何图层配置了roamConfig且active=true，会强制每帧重绘所有图层。

**检查**：查看图层配置中的roamConfig字段

### 陷阱3：effectLinkedSlices开启
视频图层的effectLinkedSlices=true时，即使视频帧未变化，也会每帧重绘。

**检查**：
```bash
curl http://192.168.1.100/api/layers | jq '.[] | select(.effectLinkedSlices==true)'
```

### 陷阱4：照片墙模式
图像图层的photoWallMode=true会持续动画。

### 陷阱5：高斯模糊过强
gaussianBlur > 5会显著增加GPU负担。
